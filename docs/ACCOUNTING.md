# Accounting module — General ledger / journal entries

A double-entry ledger sitting alongside the existing P&L aggregation
(`/reports/pl`), built for audit trail and Myanmar IRD filing.

- **Engine** — `src/lib/journal.ts` (posting engine + chart of accounts,
  `CORE_ACCOUNTS`) and `src/lib/journal-postings.ts` (one `post*()` function
  per business event).
- **Chart of accounts** — `Account` model. Fixed core accounts seeded on
  first use: Cash on Hand, Bank Account, Inventory, Accounts Payable
  (Supplier), Employee Advances Receivable, Sales Revenue, Discounts &
  Allowances, Cost of Goods Sold, Salaries & Wages Expense, Fines &
  Deductions Recovered — plus one auto-provisioned expense account per
  `ExpenseCategory` row (5100+ range, linked via
  `Account.linkedExpenseCategoryId`). No Tax Payable account — `Order` has
  no commercial-tax field today; add one before extending the journal to
  cover tax if that becomes a requirement.
- **Posting points**:
  - Order paid (`src/lib/orders.ts` `createOrder`, POS — immediate; and
    `accounting/receivable/actions.ts` `confirmPaymentProof`, online orders
    once a transfer proof is confirmed) — posts revenue + COGS atomically.
  - Supplier invoice submitted with a finalized total (`pos/invoices/actions.ts`
    `submitInvoice`) — Dr Inventory / Cr Accounts Payable.
  - Supplier invoice paid (`accounting/payable/actions.ts`
    `recordInvoicePayment`).
  - Expense entry (`accounting/expenses/actions.ts` `createExpense` — posts
    immediately, this app has no confirm/pay accrual lifecycle).
  - Salary advance given (`hr/advances/actions.ts` `createAdvance`) and
    payroll lock (`hr/payroll/[yearMonth]/actions.ts` `lockPayroll`).
- **Idempotency** — `JournalEntry` has a unique `(sourceType, sourceId)`;
  `postEntry()` no-ops if that source has already been posted, and throws on
  an unbalanced entry (debits ≠ credits — a bug, never swallowed).
- **Backfill** — `scripts/backfill-journal.ts`, run once via
  `npx tsx scripts/backfill-journal.ts` (safe to re-run). Ends with a global
  debit=credit balance check.
- **UI / export** — `/reports/journal` (date-range filtered, same
  `getDateRange` convention as `/reports/pl`); `/reports/journal/export`
  streams an `.xlsx` (via `exceljs`) with a General Journal sheet and a
  Trial Balance sheet.
- **Scope limit (deliberate)** — a General Journal feeding the P&L, not a
  full balance-sheet/GL system: no Equity/Retained-Earnings closing entries,
  no fixed-asset/depreciation tracking.

## Schema changes require a real migration here

This repo runs `prisma migrate deploy` on container start (see
`docs/DEPLOY.md`), **not** `db push`. Any further schema change to
`Account`/`JournalEntry`/`JournalLine` needs a generated migration
(`npx prisma migrate dev --name <change>`) committed alongside it — pushing
a schema change without a matching migration file will not apply on deploy.

The `add_journal_entries` migration that introduced these tables was
hand-written (no local Postgres was reachable in the environment that built
it) rather than generated by `prisma migrate dev`. It's a purely additive
change (new enum + 3 new tables, no ALTERs on existing columns), reviewed
against the schema, but **test it on UAT before trusting it on prod** —
verify `prisma migrate deploy` applies cleanly and `npx prisma generate`
matches the resulting DB shape.

## Adding a new posting

1. Add a `post*()` function to `src/lib/journal-postings.ts`. Pick a
   `sourceType` string unique to that business event.
2. Call it inside the existing action's `$transaction`, alongside the state
   change it accompanies.
3. Add the same event to `scripts/backfill-journal.ts`.
4. New account needed? Add it to `CORE_ACCOUNTS` in `journal.ts`.
